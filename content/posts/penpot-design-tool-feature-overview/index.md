<p xmlns="http://www.w3.org/1999/xhtml">Aamu.app includes a Designs section that opens a team Penpot workspace. Penpot is an open-source design and prototyping tool, and in Aamu it is used as the design surface for teams that want UI mockups, diagrams, wireframes, visual assets, or prototypes close to the rest of their project work.</p><p xmlns="http://www.w3.org/1999/xhtml">The integration is intentionally simple: Aamu gives the team a Designs entrypoint and handles the Penpot team/profile authentication flow. The actual design work happens in Penpot.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What Penpot is</h2><p xmlns="http://www.w3.org/1999/xhtml">Penpot is a browser-based design tool built around open web standards such as SVG. It is often compared with tools like Figma, but it is open source and designed to work well for both designers and developers.</p><p xmlns="http://www.w3.org/1999/xhtml">You can use Penpot for:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>UI mockups,</p></li><li><p>wireframes,</p></li><li><p>prototypes,</p></li><li><p>design systems,</p></li><li><p>icons and visual assets,</p></li><li><p>layout exploration, and</p></li><li><p>developer-friendly design handoff.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Because Penpot uses web-native concepts, it is a good fit for product teams that want design material to stay understandable to developers.</p><h2 xmlns="http://www.w3.org/1999/xhtml">How Aamu integrates it</h2><p xmlns="http://www.w3.org/1999/xhtml">In Aamu, open the <strong>Designs</strong> section. From there, Aamu provides a button to open the team's Penpot dashboard.</p><p xmlns="http://www.w3.org/1999/xhtml">Behind the scenes, Aamu handles the connection between the Aamu company and the Penpot team. When a user opens Designs, Aamu can create or verify the Penpot profile, join the user to the company Penpot team, and set the authentication token so the user can continue into Penpot.</p><p xmlns="http://www.w3.org/1999/xhtml">That means the team does not have to treat design as a completely separate login island. The design workspace is reachable from the same Aamu navigation where the team already uses Tasks, Docs, Databases, Forms, Helpdesk, Emails, and Meetings.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What belongs in Designs</h2><p xmlns="http://www.w3.org/1999/xhtml">Use Designs for visual work that benefits from a real design canvas:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>new product screens,</p></li><li><p>landing page layouts,</p></li><li><p>customer-facing UI flows,</p></li><li><p>admin interface plans,</p></li><li><p>mobile app concepts,</p></li><li><p>brand or icon exploration, and</p></li><li><p>visual explanations that are easier to draw than describe.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Use Aamu Docs for written specifications and decisions. Use Tasks for implementation work. Use Penpot when the question is visual: layout, hierarchy, components, flow, affordances, and design detail.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Designs and the rest of Aamu</h2><p xmlns="http://www.w3.org/1999/xhtml">The best workflow is not to put every product decision into the design file. Keep each kind of work in the right place:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Docs</strong> hold product requirements, copy, decisions, and design rationale.</p></li><li><p><strong>Tasks</strong> hold implementation and review work.</p></li><li><p><strong>Meetings</strong> hold design reviews and customer calls.</p></li><li><p><strong>Helpdesk</strong> shows customer feedback that can inspire design changes.</p></li><li><p><strong>Penpot</strong> holds the visual exploration and prototype.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">This keeps the design file lighter and the project history easier to follow. A design can answer "what should it look like?", while Docs and Tasks explain "why are we doing it?" and "who will build it?"</p><h2 xmlns="http://www.w3.org/1999/xhtml">For developers</h2><p xmlns="http://www.w3.org/1999/xhtml">Penpot is useful for developers because it exposes design information in a web-friendly way. Developers can inspect structure, spacing, colors, typography, and assets without needing a proprietary desktop workflow.</p><p xmlns="http://www.w3.org/1999/xhtml">For Aamu teams, this is especially useful when a design becomes implementation work. A Doc can describe the feature, a Penpot file can show the interface, and Tasks can track the actual build.</p><h2 xmlns="http://www.w3.org/1999/xhtml">For non-designers</h2><p xmlns="http://www.w3.org/1999/xhtml">You do not need a dedicated designer to get value from Designs. Penpot is also useful for rough diagrams, quick wireframes, and "this is what I mean" sketches.</p><p xmlns="http://www.w3.org/1999/xhtml">That matters in small teams. Sometimes a rough visual is faster and clearer than a long written explanation. It can also prevent misunderstandings before implementation starts.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What Aamu does not replace</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu does not try to reimplement all of Penpot inside the normal Aamu UI. The Designs section is the bridge into Penpot, while Penpot remains the design tool.</p><p xmlns="http://www.w3.org/1999/xhtml">That is a useful boundary. Aamu is where the team coordinates work and knowledge. Penpot is where the team creates and reviews design files.</p><h2 xmlns="http://www.w3.org/1999/xhtml">A practical workflow</h2><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Create a Doc for the product idea or feature brief.</p></li><li><p>Open Designs and create the related Penpot design.</p></li><li><p>Use Penpot for wireframes, screens, or prototypes.</p></li><li><p>Review the design in a Meeting or in project discussion.</p></li><li><p>Create Tasks for implementation and follow-up.</p></li><li><p>Update the Doc when decisions change.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">This keeps design connected to the work without forcing every detail into one tool.</p><h2 xmlns="http://www.w3.org/1999/xhtml">When Designs works best</h2><p xmlns="http://www.w3.org/1999/xhtml">Use Aamu Designs when the team needs a shared design space but still wants the rest of the product work in Aamu. It is especially useful for small teams that do not want design, tasks, docs, meetings, and support feedback scattered across unrelated systems.</p><p xmlns="http://www.w3.org/1999/xhtml">Penpot remains the specialized design tool. Aamu gives it a home in the workspace.</p>
