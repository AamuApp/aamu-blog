<p xmlns="http://www.w3.org/1999/xhtml">Aamu.app Docs are for the written parts of team work: internal notes, customer instructions, meeting summaries, policies, project plans, support knowledge, sales notes, technical references, and the structured writing that should stay close to tasks, Helpdesk tickets, emails, meetings, files, and databases.</p><p xmlns="http://www.w3.org/1999/xhtml">The important point is that a Doc is not only a text page. In Aamu, a Doc is a collaborative workspace object. It can be edited in a rich editor, linked to other objects, discussed in comments, shared as a public read-only version, restored from snapshots, searched, used by Team Brain, and managed through the API.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Docs in the workspace</h2><p xmlns="http://www.w3.org/1999/xhtml">Docs live inside projects, and the Docs list can show public documents, the current user's drafts, deleted documents, and project-specific document lists. Docs can be tagged, filtered through search, and surfaced in unread/unseen workflows when they matter to the team.</p><p xmlns="http://www.w3.org/1999/xhtml">That makes Docs useful for more than long-form writing. A Doc can be the explanation behind a task, the source material for a support answer, the notes from a customer meeting, the specification behind a database row, or the maintained policy that Team Brain should retrieve later.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Collaborative rich-text editing</h2><p xmlns="http://www.w3.org/1999/xhtml">The editor is built on a collaborative ProseMirror/TipTap document model. Aamu tracks save state, connection state, document versions, live editing presence, selected word and character counts, and document updates as structured editor steps.</p><p xmlns="http://www.w3.org/1999/xhtml">For everyday writing, that means the editor behaves like a modern rich-text document surface: type naturally, select text to format it, drag blocks around, insert media or structured blocks, and keep writing without switching to a separate tool.</p><h2 xmlns="http://www.w3.org/1999/xhtml">SlashMenu: everything you can insert</h2><p xmlns="http://www.w3.org/1999/xhtml">Typing <code>/</code> opens the Docs SlashMenu. It is the fastest way to add blocks, formatting, media, links, and AI help from inside the editor.</p><p xmlns="http://www.w3.org/1999/xhtml">The current SlashMenu includes these insert commands:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>AI</strong>: open the AI dialog with the current selection or document context.</p></li><li><p><strong>Internal link</strong>: search Aamu objects and insert an internal workspace link.</p></li><li><p><strong>Blockquote</strong>: add a quoted block.</p></li><li><p><strong>Code Block</strong>: insert a syntax-highlighted code block.</p></li><li><p><strong>Table</strong>: insert a table, starting with a 3 x 3 grid.</p></li><li><p><strong>2 columns</strong>: insert a two-column layout section.</p></li><li><p><strong>Drawing Canvas</strong>: add an embedded drawing canvas.</p></li><li><p><strong>Image, audio or video</strong>: add a media placeholder and upload or insert media.</p></li><li><p><strong>Share a file</strong>: search project files and insert a public file link when shared-file insertion is enabled.</p></li><li><p><strong>YouTube video</strong>: paste a YouTube URL and embed the video.</p></li><li><p><strong>JavaScript component</strong>: insert a custom browser-side component block.</p></li><li><p><strong>Custom CSS</strong>: add custom CSS for richer document rendering.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The same menu also includes formatting commands:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Heading 1</strong>, <strong>Heading 2</strong>, and <strong>Heading 3</strong>.</p></li><li><p><strong>Bold</strong> and <strong>Italic</strong>.</p></li><li><p><strong>Bullet List</strong>, <strong>Numbered List</strong>, and <strong>Task List</strong>.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The practical effect is that the Doc can become a structured working surface: a policy with tables, a meeting note with task lists, a customer page with linked files, a technical note with code blocks, or a richer internal page with embeds and custom rendering.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Selection tools and keyboard-friendly formatting</h2><p xmlns="http://www.w3.org/1999/xhtml">When text is selected, the bubble menu provides quick formatting actions: H1, H2, H3, bold, italic, underline, strikethrough, inline code, blockquote, link, and AI help for the selected text.</p><p xmlns="http://www.w3.org/1999/xhtml">Common shortcuts are also supported in the toolbar hints, such as Ctrl+B for bold, Ctrl+I for italic, Ctrl+U for underline, Ctrl+L for setting a link, and heading shortcuts for H1-H3.</p><h2 xmlns="http://www.w3.org/1999/xhtml">AI inside Docs</h2><p xmlns="http://www.w3.org/1999/xhtml">Docs are part of Aamu's AI workspace, not a separate writing island. The SlashMenu AI command and the selection bubble AI action can open an AI dialog with either selected text or the broader document context.</p><p xmlns="http://www.w3.org/1999/xhtml">That makes common writing work faster: summarize a section, rewrite selected text, draft a follow-up paragraph, turn notes into a clearer checklist, or ask a question about the content currently in front of you.</p><p xmlns="http://www.w3.org/1999/xhtml">Docs also connect to Team Brain. Maintained Docs can become the knowledge that Aamu retrieves later for Helpdesk answers, email drafts, AI commands, and internal questions. That is why writing good Docs improves more than the document itself: it improves the shared context available to the whole workspace.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Internal links and hover context</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu Docs can link to internal workspace objects. The internal link command opens a search picker so the writer can connect a Doc to tasks, projects, meetings, support tickets, files, and other workspace context.</p><p xmlns="http://www.w3.org/1999/xhtml">When internal links are used, Aamu can show hovercards and embedded context around the editor. This keeps a document connected to the work it describes instead of becoming a disconnected page.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Comments, mentions, tags, and project context</h2><p xmlns="http://www.w3.org/1999/xhtml">Docs support compact comment sidebars, unread counts, file attachments in comments, and search inside comments. They also support project placement, tags, and mentions when the editor is opened in a project context.</p><p xmlns="http://www.w3.org/1999/xhtml">That turns a Doc into a place where the team can discuss the material without moving the conversation to another app. The written content and the discussion around it stay together.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Sharing Docs publicly</h2><p xmlns="http://www.w3.org/1999/xhtml">A Doc can be shared publicly in read-only mode. Sharing creates a unique hidden URL that can be sent to someone outside the workspace.</p><p xmlns="http://www.w3.org/1999/xhtml">The shared version is snapshot-based. When the Doc is shared, Aamu stores the current HTML in a document snapshot and uses that snapshot for the public link. The share dialog shows both HTML and PDF links:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">/shared/doc/html/SHARE_ID
/shared/doc/pdf/SHARE_ID/Document-title.pdf</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Aamu also tracks how many times the shared document has been viewed. If the original Doc changes after sharing, the UI can show that a new shared version is available. The owner can then publish the new version deliberately instead of changing the public version by accident.</p><p xmlns="http://www.w3.org/1999/xhtml">Sharing can also be stopped. When sharing is stopped, the Doc no longer exposes that public read-only version.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Snapshots and recovery</h2><p xmlns="http://www.w3.org/1999/xhtml">Snapshots are useful beyond public sharing. Aamu can store document snapshots and show them in the File menu. The snapshot dialog lets the user preview stored versions, use a range control when there are multiple snapshots, and select a snapshot to restore.</p><p xmlns="http://www.w3.org/1999/xhtml">When a snapshot is selected, Aamu creates a snapshot of the current content before replacing it. That gives the user a way back from the restore operation itself.</p><p xmlns="http://www.w3.org/1999/xhtml">The Docs API also uses this safety model. When an API HTML update replaces existing Doc content through the editor step pipeline, Aamu creates a snapshot before applying the new content.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Page setup, print, and shared themes</h2><p xmlns="http://www.w3.org/1999/xhtml">The File menu includes page setup for documents that need printable or shareable presentation. Page setup supports units, orientation, margins, paper size, and the theme used in the shared document view.</p><p xmlns="http://www.w3.org/1999/xhtml">Docs can also be printed directly from the editor. For shared documents, Aamu can render the selected theme and expose the public HTML/PDF version from the shared snapshot.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Drafts, publishing, and deletion</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu separates draft and public Docs. A user's draft can remain private to the owner until it is published. Owners can publish a draft, unpublish a public Doc back to draft status, or delete a Doc from the File menu.</p><p xmlns="http://www.w3.org/1999/xhtml">This is different from public share links. Publishing controls whether the Doc is part of the team's normal public Docs list inside Aamu. Sharing creates an external read-only link for a specific snapshot.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Docs and databases</h2><p xmlns="http://www.w3.org/1999/xhtml">Docs are especially useful when paired with Aamu Databases. Database rows can reference one or more Docs, and Docs can explain the context behind structured records such as customers, deals, projects, policies, or product notes.</p><p xmlns="http://www.w3.org/1999/xhtml">For example, a CRM row can keep structured fields for stage, value, owner, and next action while linking to Docs for meeting notes, proposals, onboarding instructions, or account plans. The structured row stays scannable, and the Doc carries the richer narrative.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Docs API</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu exposes Docs through the Team API. Integrations can list project Docs, create a Doc from a title and HTML, fetch one Doc, and update its title or HTML.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">GET   /api/v1/docs/
POST  /api/v1/docs/
GET   /api/v1/docs/DOC_ID
PATCH /api/v1/docs/DOC_ID</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Send <code>x-api-key</code> and <code>x-project-id</code>. Write calls can set <code>x-aamu-actor</code>. Use Docs read permission for list/get and Docs write permission for create/update.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">POST /api/v1/docs/
x-api-key: YOUR_API_KEY
x-project-id: YOUR_PROJECT_ID
Content-Type: application/json

{
  "title": "Customer onboarding notes",
  "html": "&lt;h1&gt;Onboarding&lt;/h1&gt;&lt;p&gt;Goals, decisions, and next steps.&lt;/p&gt;"
}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">API-created Docs are native Aamu Docs, not detached HTML blobs. They remain searchable and linkable and can participate in comments, Team Brain, database document fields, and the normal editor workflow. For images or other uploaded assets, use the Files API to prepare and complete the upload, then place the returned browser URL in the Doc HTML.</p><h2 xmlns="http://www.w3.org/1999/xhtml">When to use Docs</h2><p xmlns="http://www.w3.org/1999/xhtml">Use Docs when the team needs maintained context, not only a quick message. Good examples include:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Support policies and answer templates that Team Brain should retrieve.</p></li><li><p>Meeting notes that should stay linked to follow-up tasks or CRM rows.</p></li><li><p>Project plans, implementation notes, onboarding material, and internal guides.</p></li><li><p>Customer-specific notes, proposals, and account plans.</p></li><li><p>Technical references with code blocks, tables, embedded media, and linked files.</p></li><li><p>Public read-only documents that need a stable shared HTML or PDF version.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">The bottom line</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu.app Docs are the workspace's maintained writing layer. They combine collaborative editing, slash-command blocks, AI help, internal links, comments, tags, sharing, snapshots, page setup, and API access.</p><p xmlns="http://www.w3.org/1999/xhtml">That combination matters because knowledge is only useful when it stays connected to the work around it. A Doc can explain the task, support the ticket, document the meeting, enrich the CRM row, feed Team Brain, and still be shared externally as a controlled snapshot when needed.</p>
