<p xmlns="http://www.w3.org/1999/xhtml">Aamu.app Tasks is the place where work becomes actionable. A task can be a simple to-do, a scheduled event, a recurring responsibility, a support follow-up, a form-submission workflow, or an API-created work item. The important idea is that tasks live in the same workspace as Docs, Databases, Forms, Emails, Helpdesk, Meetings, and AI.</p><p xmlns="http://www.w3.org/1999/xhtml">This is a current feature overview of Tasks: how tasks are organized, what you can put inside a task, and how Tasks connect to the rest of Aamu.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Projects and task lists</h2><p xmlns="http://www.w3.org/1999/xhtml">Tasks belong to projects. You can work inside one project, or use the combined all-projects view when you want to see work across the whole team.</p><p xmlns="http://www.w3.org/1999/xhtml">The task header gives you the main controls:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>search inside the selected project or across projects,</p></li><li><p>create a new task,</p></li><li><p>sort the current list,</p></li><li><p>filter by user, tag, or custom status field,</p></li><li><p>switch between task views, and</p></li><li><p>move between active, overdue, running, completed, and archived work.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">This makes Tasks useful both as a personal work queue and as a project-level operations board.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Task views</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu has multiple ways to look at the same tasks.</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>List</strong> is the dense view for scanning, sorting, filtering, and batch work.</p></li><li><p><strong>Taskboard</strong> is the kanban-style view for moving work visually.</p></li><li><p><strong>Calendar</strong> shows scheduled tasks and events by time.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The same task can appear in different views depending on how you work. A task with a due date is still the same task when you see it in the list, board, or calendar.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Sorting and filtering</h2><p xmlns="http://www.w3.org/1999/xhtml">The list view can be sorted by useful operational signals such as updated time, created time, starting date, due date, star count, comment count, completed time, and total time used.</p><p xmlns="http://www.w3.org/1999/xhtml">Filtering is where project structure becomes useful. You can filter by assignee, tags, and project custom status fields. For example, a project can have a custom status field for "Stage" or "Priority", and that field can be used to filter the task list.</p><p xmlns="http://www.w3.org/1999/xhtml">Search also supports structured queries such as assignee and status-style filters, so larger task lists do not need to become messy.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What a task contains</h2><p xmlns="http://www.w3.org/1999/xhtml">A task can contain much more than a title. A typical Aamu task can have:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>a rich-text description,</p></li><li><p>comments,</p></li><li><p>assignees,</p></li><li><p>tags,</p></li><li><p>files and attachments,</p></li><li><p>todos inside the task,</p></li><li><p>start and due dates,</p></li><li><p>reminders,</p></li><li><p>repetition,</p></li><li><p>custom fields, and</p></li><li><p>internal links to other Aamu content.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The description and comments use the same workspace thinking as the rest of Aamu: the task is not only a checkbox, it is where the context for the work can live.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Dates, events, and recurring work</h2><p xmlns="http://www.w3.org/1999/xhtml">A task can be a simple due-date item, or it can behave more like an event with both a start and an end date. Aamu shows due tasks, upcoming events, overdue tasks, and running work separately so time-sensitive work is easier to notice.</p><p xmlns="http://www.w3.org/1999/xhtml">Tasks can also repeat daily, weekly, monthly, or yearly. This is useful for recurring operational responsibilities: weekly reporting, monthly checks, renewal reviews, bookkeeping reminders, and regular customer follow-ups.</p><p xmlns="http://www.w3.org/1999/xhtml">When a task has a due date, Aamu can also produce an iCalendar file for it. Projects can additionally be connected to Google Calendar in project settings.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Custom task fields</h2><p xmlns="http://www.w3.org/1999/xhtml">Project custom fields make Tasks adapt to the way a team works. Current custom field types include:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Text,</p></li><li><p>Long Text,</p></li><li><p>Number,</p></li><li><p>Status,</p></li><li><p>Time &amp; Date,</p></li><li><p>User, and</p></li><li><p>Location.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Status fields are especially useful because they can become filters and lightweight process stages. For example, a project can track work by "New", "Waiting", "In progress", and "Done", or by a custom sales/support/production stage.</p><p xmlns="http://www.w3.org/1999/xhtml">Custom fields can also be shown in task cards, which makes the list and board views easier to scan.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Bulk actions</h2><p xmlns="http://www.w3.org/1999/xhtml">When you select multiple tasks, Aamu can handle common batch operations. You can assign users, set due dates, close or reopen tasks, and archive or unarchive tasks depending on the current view.</p><p xmlns="http://www.w3.org/1999/xhtml">This matters when Tasks is used for operations, not only personal planning. A form or database automation might create many tasks, and the team still needs fast ways to triage them.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Tasks connected to Forms and Databases</h2><p xmlns="http://www.w3.org/1999/xhtml">Tasks connect naturally to Aamu Forms and Databases. A form submission can create a database row, and a database automation can create a task from that row.</p><p xmlns="http://www.w3.org/1999/xhtml">For example:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>a contact form can create a sales follow-up task,</p></li><li><p>a bug report form can create a triage task,</p></li><li><p>a survey response can create a review task, and</p></li><li><p>an onboarding request can create an implementation task.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The database keeps the structured response data. The task carries the work forward.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Tasks connected to Helpdesk and Emails</h2><p xmlns="http://www.w3.org/1999/xhtml">Support and email work often turns into tasks. A customer asks for something, a support ticket reveals follow-up work, or an email thread needs a concrete next step. In Aamu, those actions can stay inside the same workspace instead of being copied into a separate task tool.</p><p xmlns="http://www.w3.org/1999/xhtml">This is one of the main reasons Tasks is central to Aamu. The task does not have to be isolated from the conversation or document that created it.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Tasks and AI</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu AI can help turn workspace context into task drafts. For example, a conversation, doc, email, or support ticket can become a proposed task with a title and body. The useful part is not just that AI writes text; it writes into the place where the work will actually happen.</p><p xmlns="http://www.w3.org/1999/xhtml">As with the rest of Aamu, the safer pattern is to use AI to prepare work and let a person decide what should be saved, assigned, or acted on.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Tasks API</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu also exposes Tasks through the API for integrations. The core endpoints are:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">GET   /api/v1/tasks/
POST  /api/v1/tasks/
GET   /api/v1/tasks/{id}
PATCH /api/v1/tasks/{id}
POST  /api/v1/tasks/{id}/comments</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Use the Tasks API when another system needs to create work in Aamu, update task state, or add comments. API keys are project-scoped, so use the correct project scope for the task workflow you are building.</p><h2 xmlns="http://www.w3.org/1999/xhtml">When Tasks works best</h2><p xmlns="http://www.w3.org/1999/xhtml">Tasks works best when it is treated as the operational layer of the workspace. Docs hold durable knowledge. Databases hold structured data. Forms collect input. Helpdesk and Emails handle communication. Tasks turn all of that into assigned, scheduled, trackable work.</p><p xmlns="http://www.w3.org/1999/xhtml">That is the reason Tasks is not just another to-do list inside Aamu.app. It is the place where the rest of the workspace becomes action.</p>
