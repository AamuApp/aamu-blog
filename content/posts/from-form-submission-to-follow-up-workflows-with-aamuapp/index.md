<p xmlns="http://www.w3.org/1999/xhtml">A form submission is rarely the end of a workflow. A demo request needs an owner. A support escalation needs a task. A job application needs review. A new lead may need a confirmation email, a CRM record, and a follow-up deadline.</p><p xmlns="http://www.w3.org/1999/xhtml">Aamu.app can keep that whole chain in one workspace. Forms collect structured input. Databases keep the source record. Automations turn matching row events into Tasks or Emails. Outbound webhooks notify external systems. CRM tables keep the relationship and its history visible after the first response.</p><p xmlns="http://www.w3.org/1999/xhtml">This article shows how those parts fit together and how to design workflows that remain understandable after the first successful test.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The workflow model</h2><p xmlns="http://www.w3.org/1999/xhtml">A practical Aamu workflow has five layers:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Input</strong>: a public form, Forms API request, GraphQL mutation, or manual database entry.</p></li><li><p><strong>Structured state</strong>: a database row containing the submitted fields and the current workflow status.</p></li><li><p><strong>Trigger</strong>: a row being inserted or a selected field changing to a configured value.</p></li><li><p><strong>Action</strong>: create an Aamu task, send an email, or call an external system through a webhook.</p></li><li><p><strong>Ongoing work</strong>: update the CRM row, complete the task, continue the email conversation, or move the row to its next stage.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">The database row is the durable center of the workflow. Notifications and tasks can change over time, but the original structured input and its status remain available for search, reporting, activity history, and later automation.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Workflow 1: form submission to follow-up task</h2><p xmlns="http://www.w3.org/1999/xhtml">Consider a demo request form with these fields:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Name</p></li><li><p>Email</p></li><li><p>Company</p></li><li><p>What the team wants to solve</p></li><li><p>Preferred follow-up date</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Publishing the form gives the team a browser-friendly input surface. Every submission becomes a row in the form’s database table. The row can also include internal fields that the public form does not expose, such as owner, status, qualification notes, next action, and related tasks.</p><h3 xmlns="http://www.w3.org/1999/xhtml">Trigger on the inserted row</h3><p xmlns="http://www.w3.org/1999/xhtml">Create a <strong>Row inserted</strong> automation for the response table. A single automation can contain more than one action:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>send a confirmation or internal notification email, and</p></li><li><p>create a follow-up task in the sales or customer project.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Map the company or request field to the task title and the longer request description to the task body. Assign the task to the person or team responsible for reviewing new requests.</p><p xmlns="http://www.w3.org/1999/xhtml">The result is a useful separation of concerns: the database row stores the lead, while the task represents the work someone must do next.</p><h3 xmlns="http://www.w3.org/1999/xhtml">The same workflow through the API</h3><p xmlns="http://www.w3.org/1999/xhtml">An authenticated integration can submit an existing form through:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">POST /api/v1/forms/FORM_ID/submissions
x-api-key: YOUR_API_KEY
x-project-id: YOUR_PROJECT_ID
Content-Type: application/json

{
  "fields": {
    "name": "Ada Lovelace",
    "email": "ada@example.com",
    "company": "Analytical Engines Ltd",
    "request": "We need a shared workflow for customer onboarding."
  }
}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The Forms API creates the same kind of database row as a browser submission, so the same public row-inserted automation runs afterward.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Workflow 2: CRM stage change to handoff</h2><p xmlns="http://www.w3.org/1999/xhtml">Some workflows should not run immediately when a row is created. A new deal may need qualification before delivery or onboarding work begins.</p><p xmlns="http://www.w3.org/1999/xhtml">In a CRM table, use a status field such as:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">new → qualified → proposal → won → onboarding</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Create a <strong>Row updated</strong> automation that watches the Stage field and runs when the new value becomes <code>won</code>. The automation can then create a handoff task in the delivery project. It runs only when the value actually changes to the configured value, not when the same value is written again.</p><p xmlns="http://www.w3.org/1999/xhtml">This gives the CRM row two jobs:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>it records the current commercial state, and</p></li><li><p>it starts the next operational workflow at the correct transition.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The new task can use the deal name as its title, copy implementation notes into the body, assign delivery users, and map database values into custom task fields.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Automations or webhooks?</h2><p xmlns="http://www.w3.org/1999/xhtml">Database automations and outbound webhooks solve related but different problems.</p><table xmlns="http://www.w3.org/1999/xhtml" style="min-width: 50px;"><colgroup><col style="min-width: 25px;" ><col style="min-width: 25px;" ></colgroup><tbody><tr><th colspan="1" rowspan="1"><p>Use</p></th><th colspan="1" rowspan="1"><p>Choose</p></th></tr><tr><td colspan="1" rowspan="1"><p>Create an Aamu task from a row</p></td><td colspan="1" rowspan="1"><p>Database automation</p></td></tr><tr><td colspan="1" rowspan="1"><p>Send through a configured Aamu project email</p></td><td colspan="1" rowspan="1"><p>Database automation</p></td></tr><tr><td colspan="1" rowspan="1"><p>Notify another application about the submission</p></td><td colspan="1" rowspan="1"><p>Outbound webhook</p></td></tr><tr><td colspan="1" rowspan="1"><p>Start custom code, a queue, or an external integration</p></td><td colspan="1" rowspan="1"><p>Outbound webhook</p></td></tr><tr><td colspan="1" rowspan="1"><p>Fetch more context or write back into Aamu</p></td><td colspan="1" rowspan="1"><p>Webhook followed by an Aamu API call</p></td></tr></tbody></table><p xmlns="http://www.w3.org/1999/xhtml">A simple rule is: use an automation when the next action belongs inside Aamu, and use a webhook when another system needs to react.</p><p xmlns="http://www.w3.org/1999/xhtml">The two can coexist. A form submission can create an internal task and also send a signed <code>form.submitted</code> webhook to an external service. The external service can verify the signature, store the delivery id for idempotency, and call the Aamu API when it needs more detail or wants to write a result back.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Provision the workflow through APIs</h2><p xmlns="http://www.w3.org/1999/xhtml">The workflow can be assembled programmatically when an integration needs repeatable setup.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">POST /api/v1/databases/
POST /api/v1/databases/DB_ID/tables/TABLE_ID/columns
POST /api/v1/databases/DB_ID/automations
POST /api/v1/graphql/
GET  /api/v1/tasks/</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Database schema calls use Database write scope. Automation definitions use the separate Automations scope. A task action additionally requires Tasks write scope for its target project, while an email action requires Emails write scope.</p><p xmlns="http://www.w3.org/1999/xhtml">For example, an integration can create a request database, add title/details/status columns, create a public row-inserted automation, and then insert rows through the generated GraphQL mutation. GraphQL row inserts run the same row-inserted automation path as Forms API submissions.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Design the database for the workflow</h2><p xmlns="http://www.w3.org/1999/xhtml">Good automation starts with understandable data. Useful workflow columns often include:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Status or stage</strong> for explicit transitions.</p></li><li><p><strong>Owner</strong> for responsibility.</p></li><li><p><strong>Next action and due date</strong> for follow-up.</p></li><li><p><strong>Source</strong> to distinguish public form, API, import, or manual entry.</p></li><li><p><strong>Related task, email, meeting, or Doc fields</strong> for connected work.</p></li><li><p><strong>External id</strong> when another system must correlate or deduplicate records.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Do not make automation depend on ambiguous free-form notes when a status field can express the decision directly. A field changing to <code>approved</code> is a safer trigger than trying to infer approval from a comment.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Keep humans in the loop where it matters</h2><p xmlns="http://www.w3.org/1999/xhtml">Not every action should happen automatically. Creating an internal review task is low risk. Sending a customer email, changing a CRM stage, or starting an external process may deserve a deliberate state transition or human approval.</p><p xmlns="http://www.w3.org/1999/xhtml">A useful pattern is:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>The form creates a row and an internal review task.</p></li><li><p>A person reviews the information and changes Status to Approved.</p></li><li><p>A row-updated automation creates the operational handoff task or sends the approved email.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">That keeps automation fast without letting raw public input immediately trigger every downstream side effect.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Test the whole chain</h2><p xmlns="http://www.w3.org/1999/xhtml">A workflow is not tested when its individual settings look correct. Test the complete path:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Submit a realistic form response or GraphQL mutation.</p></li><li><p>Confirm that the database row contains the expected values.</p></li><li><p>Confirm that the automation trigger matches the correct table and event.</p></li><li><p>Confirm that the task appears in the correct project and task list.</p></li><li><p>Confirm that email bindings produced the intended recipient, subject, and body.</p></li><li><p>If using webhooks, verify the signature and inspect the delivery log.</p></li><li><p>Repeat the update with the same value and confirm that a row-updated automation does not run twice.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">Use clearly named test records and clean up temporary automations afterward. Keep test database rows when they are useful as examples, or mark them so they do not affect real reporting.</p><h2 xmlns="http://www.w3.org/1999/xhtml">A practical architecture</h2><p xmlns="http://www.w3.org/1999/xhtml">For many small teams, the complete workflow can stay pleasantly simple:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">Form or API
    ↓
Database row
    ↓
Automation ──→ Task or Email inside Aamu
    │
    └────────→ Signed webhook to an external system
                    ↓
               Optional Aamu API call
                    ↓
              CRM row or follow-up update</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The strength of this model is not that every step is automatic. It is that input, structured state, internal work, customer communication, and external integrations share clear boundaries.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Related guides</h2><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/creating-a-contact-form-with-aamuapp/">Creating a contact form with Aamu.app</a></p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/database-automations-with-aamuapp/">Database automations with Aamu.app</a></p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/outbound-webhooks-in-aamuapp-real-time-events-tasks-helpdesk-email/">Outbound webhooks in Aamu.app</a></p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/how-to-use-aamuapp-as-a-small-team-crm/">How to use Aamu.app as a small-team CRM</a></p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/building-with-the-aamu-api-from-tasks-to-docs-and-graphql/">Building with the Aamu API</a></p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">The bottom line</h2><p xmlns="http://www.w3.org/1999/xhtml">Forms collect the input. Tasks, Emails, and CRM tables keep the workflow visible after the first event.</p><p xmlns="http://www.w3.org/1999/xhtml">That is enough to build useful lead handling, onboarding, support escalation, recruiting, approvals, and internal request workflows without turning every process into a separate application.</p>
