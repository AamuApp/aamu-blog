<p xmlns="http://www.w3.org/1999/xhtml">Email setup is one of the more technical parts of Aamu.app because it touches domains, DNS records, mailboxes, and project settings. The good news is that the flow is understandable once you separate it into three layers: the team owns the email domain, the project chooses which domain it uses, and each user configures their own mailbox.</p><p xmlns="http://www.w3.org/1999/xhtml">The same domain setup is also useful for Helpdesk, because Helpdesk email tickets need a configured email route too. If you plan to use both Emails and Helpdesk, set up the email domain carefully once and reuse it where needed.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The setup flow</h2><p xmlns="http://www.w3.org/1999/xhtml">The high-level flow is:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Add an email domain in Team settings.</p></li><li><p>Choose whether Aamu handles the mail server side or whether you use a third-party IMAP/SMTP server.</p></li><li><p>Add the required DNS records at your domain provider.</p></li><li><p>Verify the domain in Aamu.</p></li><li><p>Attach the verified domain to a project.</p></li><li><p>Let users create their project mailbox.</p></li><li><p>Start sending and receiving email from Aamu.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">The details matter, but this is the shape of the process.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Email domains</h2><p xmlns="http://www.w3.org/1999/xhtml">An email domain is configured at the team level. A domain can then be attached to a project that needs email. In practice, this means your team first proves that it controls the domain, and then projects can use it for mailboxes and email workflows.</p><p xmlns="http://www.w3.org/1999/xhtml">One project can use one email domain for project email. Within that project, each user can have their own mailbox on that domain. Those mailboxes are personal: other users do not automatically see another user's personal email account, messages, or contacts.</p><p xmlns="http://www.w3.org/1999/xhtml">To add a domain, open the profile or team menu, go to Team settings, and find Email domains. Add the domain you want to use.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Aamu mail server or third-party mail server?</h2><p xmlns="http://www.w3.org/1999/xhtml">When adding the domain, Aamu can support different mail setups.</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Aamu.app mail server</strong>: Aamu provides the mail server side for the domain. You add the DNS records Aamu gives you, verify the domain, and create mailboxes in Aamu.</p></li><li><p><strong>Third-party IMAP/SMTP server</strong>: you keep the mailbox with another provider and connect Aamu to it with IMAP for receiving and SMTP for sending.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The Aamu-hosted option is usually the simpler all-in-one route. The third-party option is useful when you already have mailboxes elsewhere and want Aamu to work with that existing setup.</p><h2 xmlns="http://www.w3.org/1999/xhtml">DNS verification</h2><p xmlns="http://www.w3.org/1999/xhtml">After adding the domain, Aamu shows the DNS records you need to add at your DNS provider. That provider might be Cloudflare, your domain registrar, your hosting provider, or another DNS service.</p><p xmlns="http://www.w3.org/1999/xhtml">The exact records shown in Aamu are the source of truth. Copy them carefully. Email domain setup commonly includes records for things like receiving mail and proving that Aamu is allowed to send mail for the domain. Depending on the setup, these may include MX, SPF, DKIM, or verification records.</p><p xmlns="http://www.w3.org/1999/xhtml">DNS changes may take time to propagate. If verification does not succeed immediately, check for typos first, then wait a little and try again. It is normal for DNS to take some time.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Attach the domain to a project</h2><p xmlns="http://www.w3.org/1999/xhtml">After the domain is verified, attach it to the project where you want to use email. Open the project settings and go to the Email settings for that project. Select the verified domain.</p><p xmlns="http://www.w3.org/1999/xhtml">This step matters because the team may have more than one project, and each project may use a different email domain or mailbox setup. Attaching the domain tells Aamu which domain this project should use for Emails and related workflows.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Create user mailboxes</h2><p xmlns="http://www.w3.org/1999/xhtml">Once the project has an email domain, users can create their project mailbox. Go to the Emails section, select the project, and open Email settings.</p><p xmlns="http://www.w3.org/1999/xhtml">Each user chooses their email address and display name for that project. For example, one user might use <code>ilkkah@example.com</code>, while another uses <code>support-person@example.com</code>. The available address depends on the configured domain and mailbox setup.</p><p xmlns="http://www.w3.org/1999/xhtml">Users can also configure personal email details such as a footer or tabs. Tabs work like folders or views for organizing messages.</p><h2 xmlns="http://www.w3.org/1999/xhtml">How Emails and Helpdesk relate</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu Emails and Aamu Helpdesk are separate workflows, but they can depend on the same email domain foundation.</p><p xmlns="http://www.w3.org/1999/xhtml">Use Emails when users need personal or project email inside Aamu. Use Helpdesk when incoming messages should become support tickets that the team can handle together. If you are setting up Helpdesk email later, this email domain setup is the groundwork.</p><p xmlns="http://www.w3.org/1999/xhtml">The main distinction is privacy and workflow:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Personal project mailboxes belong to users.</p></li><li><p>Helpdesk mailboxes are used for shared support workflows.</p></li><li><p>Both need correct domain and mailbox configuration.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">Testing the setup</h2><p xmlns="http://www.w3.org/1999/xhtml">After creating a mailbox, test both directions.</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Send an email from Aamu to an outside address you control.</p></li><li><p>Reply from that outside address.</p></li><li><p>Confirm that the incoming reply appears in Aamu.</p></li><li><p>Check whether the sender, display name, and footer look correct.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">If sending works but receiving does not, check the receiving-side DNS and IMAP settings. If receiving works but sending does not, check the sending-side DNS and SMTP settings. If neither works, start with domain verification and project attachment.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Troubleshooting checklist</h2><p xmlns="http://www.w3.org/1999/xhtml">If email setup gets stuck, check these:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>The domain is added in Team settings.</p></li><li><p>All DNS records shown by Aamu have been added exactly.</p></li><li><p>DNS changes have had time to propagate.</p></li><li><p>The domain verifies successfully in Aamu.</p></li><li><p>The verified domain is attached to the correct project.</p></li><li><p>The user has created a mailbox in the Emails settings for that project.</p></li><li><p>For third-party mail servers, IMAP and SMTP host, port, username, and password settings are correct.</p></li><li><p>The mailbox address and display name are set as expected.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">Done</h2><p xmlns="http://www.w3.org/1999/xhtml">After the domain is verified, attached to the project, and the user mailbox is created, you can send and receive email in Aamu.app.</p><p xmlns="http://www.w3.org/1999/xhtml">The practical benefit is that email no longer lives apart from the team's work. Messages can be connected to tasks, contacts, customer workflows, Helpdesk setup, and the rest of the Aamu workspace.</p>
