<p xmlns="http://www.w3.org/1999/xhtml">Aamu.app database automations let a database row start follow-up work automatically. They are especially useful with forms: a public form collects the data, the response becomes a database row, and an automation can send an email or create a task when that row is inserted.</p><p xmlns="http://www.w3.org/1999/xhtml">This article uses the contact form idea as an example, but the same pattern works for survey responses, onboarding requests, bug reports, event registrations, lead forms, and internal request queues.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The current automation model</h2><p xmlns="http://www.w3.org/1999/xhtml">A database automation belongs to one Aamu database. Inside the automation you choose:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>the automation name,</p></li><li><p>the triggering type,</p></li><li><p>the triggering table,</p></li><li><p>one or more actions, and</p></li><li><p>whether the automation is draft or public.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The current database trigger is <strong>Row inserted</strong>. In other words, the automation runs when a new row is added to the selected table. That can happen through a published Aamu form, the Forms API, a database API write, or a manual database entry.</p><p xmlns="http://www.w3.org/1999/xhtml">The current actions are:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Send an email</strong></p></li><li><p><strong>Create a task</strong></p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">You can add more than one action to the same automation. Actions can be inserted before, between, or after existing actions, and removed when they are no longer needed. For example, one new contact-form submission can send a confirmation email and create a follow-up task for the sales team.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Open automations from the database</h2><p xmlns="http://www.w3.org/1999/xhtml">Open the database that contains the table you want to automate. From the database menu, open <strong>Automations</strong>, then create a new automation.</p><p xmlns="http://www.w3.org/1999/xhtml">If you are coming from a form, remember that form responses are stored in a database table. The form's Responses view links back to that database. That is usually the easiest place to find the right database and table.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Choose the trigger table</h2><p xmlns="http://www.w3.org/1999/xhtml">Give the automation a descriptive name, such as:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">Notify us about new contact form submissions
Create follow-up task from onboarding requests
Send survey thank-you email</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Set the triggering type to <strong>Row inserted</strong>, then choose the table that receives new rows.</p><p xmlns="http://www.w3.org/1999/xhtml">After you select the table, Aamu shows the available template bindings. These are placeholders based on the database fields. A field might be shown as something like:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">{{email}}
{{name}}
{{message}}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">You can use these bindings in email templates and, depending on the action, map database fields into task data.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Action: send an email</h2><p xmlns="http://www.w3.org/1999/xhtml">The email action sends through an Aamu email account configured for a project. That means email needs to be set up before this action can run.</p><p xmlns="http://www.w3.org/1999/xhtml">When configuring the action, choose the email address to use for sending. Then create a new email template or select a previously created template. Existing templates can also be edited from the automation. The template can use database bindings in the recipient, subject, and body.</p><p xmlns="http://www.w3.org/1999/xhtml">For a contact form, the template might use:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">To: support@example.com
Subject: New contact form message from {{name}}

Message:
{{message}}

Reply to:
{{email}}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">If a field may be empty, templates can use a default value form for bindings:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">{{company|No company provided}}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Use the email action for notifications, confirmations, and lightweight handoffs. For real work that needs ownership and status, create a task too.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Action: create a task</h2><p xmlns="http://www.w3.org/1999/xhtml">The task action creates an Aamu task when a new database row is inserted. Choose the project where the task should be created, then choose the user who should be assigned.</p><p xmlns="http://www.w3.org/1999/xhtml">You can map database fields into the task:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>a database field for the task title,</p></li><li><p>a database field for the task body, and</p></li><li><p>database fields for custom task fields when the target project has them.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">This is useful when a submitted form needs a real workflow. For example:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>a demo request creates a sales task,</p></li><li><p>a bug report creates a triage task,</p></li><li><p>a job application creates a recruiting task, or</p></li><li><p>an onboarding request creates an implementation task.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The database row remains the source data. The task is the follow-up work.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Draft vs public</h2><p xmlns="http://www.w3.org/1999/xhtml">New automations start as draft. A draft automation does not run. This is deliberate: you can configure the trigger, actions, templates, and mappings without affecting live data.</p><p xmlns="http://www.w3.org/1999/xhtml">Set the automation to <strong>Public</strong> when it is ready. Only public automations run when rows are inserted.</p><p xmlns="http://www.w3.org/1999/xhtml">If you need to pause the workflow, switch the automation back to draft. This is a simple way to turn the automation off without deleting its configuration.</p><p xmlns="http://www.w3.org/1999/xhtml">The user who creates the automation owns its configuration. Other users can view the automation, but editing controls are available to the owner.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Test the automation</h2><p xmlns="http://www.w3.org/1999/xhtml">When an automation is public and has actions, Aamu shows a test action control. Use it before relying on the workflow.</p><p xmlns="http://www.w3.org/1999/xhtml">For a form-based workflow, also test the real path:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Open the public form.</p></li><li><p>Submit a realistic test response.</p></li><li><p>Confirm that a new row appears in the database table.</p></li><li><p>Confirm that the email was sent, if the automation sends email.</p></li><li><p>Confirm that the task was created in the correct project, if the automation creates a task.</p></li><li><p>Check that the field bindings produced readable text.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">Testing with a real submission catches the full chain: public form, database row creation, automation trigger, email configuration, and task mapping.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Troubleshooting checklist</h2><p xmlns="http://www.w3.org/1999/xhtml">If the automation does not run, check these:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>The automation status is Public, not Draft.</p></li><li><p>The trigger table is the same table receiving the new row.</p></li><li><p>The row is actually being inserted, not only edited.</p></li><li><p>The action has all required settings.</p></li><li><p>The sending email account is configured for the selected project.</p></li><li><p>The email domain is attached to the project if email sending depends on it.</p></li><li><p>The task action points to the right project.</p></li><li><p>The selected assignee still has access to that project.</p></li><li><p>The template bindings match the fields in the triggering table.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">Database automations and outbound webhooks</h2><p xmlns="http://www.w3.org/1999/xhtml">Database automations are designed for internal follow-up after a row is inserted: send an email, create a task, or do both. If another system needs to react to Aamu events, use <a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/outbound-webhooks-in-aamuapp-real-time-events-tasks-helpdesk-email/">outbound webhooks</a>. Webhooks cover a broader range of workspace events and send signed event data to an external HTTPS endpoint.</p><p xmlns="http://www.w3.org/1999/xhtml">A simple rule of thumb is: use a database automation when the next action should happen inside Aamu, and use an outbound webhook when the next action belongs in another system.</p><h2 xmlns="http://www.w3.org/1999/xhtml">A practical pattern</h2><p xmlns="http://www.w3.org/1999/xhtml">A good Aamu automation usually has this shape:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>A form or API call creates a database row.</p></li><li><p>The database row stores the structured data.</p></li><li><p>A database automation sends a notification or creates a task.</p></li><li><p>The team handles the actual work in Tasks, Emails, Helpdesk, or the database.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">That is the strength of keeping forms, databases, email, and tasks in the same workspace. The submitted data does not disappear into a notification. It stays available as a row, and the automation turns it into the next action.</p>
