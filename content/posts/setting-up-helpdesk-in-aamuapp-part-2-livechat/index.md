<p xmlns="http://www.w3.org/1999/xhtml">Aamu Helpdesk can receive support requests by email, but it can also talk to visitors while they are still on your website. That is what Aamu Livechat is for: a public chat widget connected to the same Helpdesk project where your team handles tickets, comments, assignments, follow-up tasks, and customer conversations.</p><p xmlns="http://www.w3.org/1999/xhtml">This article continues from <a target="_blank" rel="noopener noreferrer nofollow" href="https://aamu.app/blog/posts/setting-up-helpdesk-in-aamuapp-part-1-emails/">Part 1: Helpdesk emails</a>. Email support is the slower, asynchronous route. Livechat is the real-time route, with an optional email fallback when nobody is available and optional AI answers when the project allows them.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Before you start</h2><p xmlns="http://www.w3.org/1999/xhtml">You need a project with Helpdesk enabled and the right users added to the Helpdesk project. If you have not done that yet, set up the Helpdesk project and users first.</p><p xmlns="http://www.w3.org/1999/xhtml">For the best customer experience, also configure Helpdesk email before adding Livechat. Livechat can fall back to an email-style message when the chat cannot be handled immediately, so the email route gives visitors another way to reach you.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Open the Livechat settings</h2><p xmlns="http://www.w3.org/1999/xhtml">Open the project settings for the project that should receive the chats, then go to the Helpdesk Livechat settings. The route looks like this:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">/project/edit/PROJECT_ID/helpdesk/livechat</code></pre><p xmlns="http://www.w3.org/1999/xhtml">This page contains the public widget code, the texts shown in the widget, and the AI settings for live chat. Team admins can edit the project-level Helpdesk Livechat settings.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Add the widget to your website</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu gives you an HTML snippet for the public website. It uses the hosted Livechat script and an <code>aamu-livechat</code> element with your Aamu host and project id.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-html">&lt;script defer src="https://st.aamu.app/livechat.js"&gt;&lt;/script&gt;
&lt;aamu-livechat
  data-username=""
  data-useremail=""
  data-host="YOUR_AAMU_SUBDOMAIN"
  data-pid="PROJECT_ID"&gt;&lt;/aamu-livechat&gt;</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Copy the snippet from Aamu instead of inventing the values manually. The project id in the snippet can be an encoded project id, so the version shown in the UI is the one to use.</p><p xmlns="http://www.w3.org/1999/xhtml">You can place the snippet in the <code>&lt;head&gt;</code> or near the end of the <code>&lt;body&gt;</code> on the pages where the chat should appear.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Passing visitor details</h2><p xmlns="http://www.w3.org/1999/xhtml">The widget supports optional visitor identity fields:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><code>data-username</code> for the visitor's name,</p></li><li><p><code>data-useremail</code> for the visitor's email address,</p></li><li><p><code>data-host</code> for your Aamu subdomain, and</p></li><li><p><code>data-pid</code> for the Helpdesk project.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">If your website already knows the visitor, you can fill in the name and email. If not, leave them empty and let the conversation collect the details when needed.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Human Livechat</h2><p xmlns="http://www.w3.org/1999/xhtml">In the normal human-support flow, a visitor opens the widget and starts a chat. Helpdesk users can answer from Aamu. The chat becomes part of the Helpdesk history, and the team can connect the conversation to a ticket or continue the support work from there.</p><p xmlns="http://www.w3.org/1999/xhtml">Each support user can also set their own Livechat identity in Helpdesk settings. This is the name visitors see when that person answers the chat.</p><p xmlns="http://www.w3.org/1999/xhtml">Use human Livechat when a visitor needs judgment, account-specific help, sales conversation, or anything where a real person should make the call.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Email fallback</h2><p xmlns="http://www.w3.org/1999/xhtml">Livechat should not become a dead end when nobody is available. The widget can move into an email-style message flow when the server or support team cannot take a real-time chat. The visitor can leave a message, and the team can handle it later through Helpdesk.</p><p xmlns="http://www.w3.org/1999/xhtml">This is why the email setup from Part 1 matters. Real-time chat is great when someone can answer now. Email fallback is what keeps the support route useful outside office hours or during busy moments.</p><h2 xmlns="http://www.w3.org/1999/xhtml">AI Livechat</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu Livechat can also use AI when the project setting <strong>Allow AI to answer live chat messages</strong> is enabled. You can find this in the project Helpdesk Livechat settings.</p><p xmlns="http://www.w3.org/1999/xhtml">When enabled, the public widget opens in chat mode and AI can answer from Team Brain knowledge sources. The project can also configure the AI name and opening message, for example using a name such as <code>Aamu AI</code>.</p><p xmlns="http://www.w3.org/1999/xhtml">This is different from Helpdesk and email reply drafts. A reply draft waits for a human to review and send it. A Livechat AI answer can be shown directly to the visitor during the conversation, so the knowledge base and escalation rules matter more.</p><p xmlns="http://www.w3.org/1999/xhtml">Use AI Livechat for common, safe, well-documented questions. Use human support for account-specific, sensitive, contractual, billing, or uncertain cases.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Team Brain as the knowledge source</h2><p xmlns="http://www.w3.org/1999/xhtml">AI Livechat should answer from knowledge that belongs to your team, not from a generic guess. In Aamu, that context comes from Team Brain knowledge sources connected to the Helpdesk project.</p><p xmlns="http://www.w3.org/1999/xhtml">Before enabling AI answers broadly, add the source material that the AI is allowed to use. Good sources include:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>public FAQ pages,</p></li><li><p>product documentation,</p></li><li><p>pricing and plan pages,</p></li><li><p>support policy docs,</p></li><li><p>onboarding instructions, and</p></li><li><p>manual FAQ snippets for answers that are not published elsewhere.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">After changing the knowledge sources, reindex them and preview retrieved sources with real customer questions. This helps you catch missing or ambiguous material before visitors depend on the answers.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Customize the widget text</h2><p xmlns="http://www.w3.org/1999/xhtml">The Livechat settings include translations and text labels for the public widget. You can edit button labels, header text, queue text, feedback text, and email panel text.</p><p xmlns="http://www.w3.org/1999/xhtml">Keep these short and concrete. The widget is small, and the visitor is usually trying to get help quickly. A clear label such as "Send" or "Close chat" is better than a long instruction.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Test the whole route</h2><p xmlns="http://www.w3.org/1999/xhtml">After saving the settings and adding the snippet to your website, test the flow as an outside visitor.</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Open the public website in a private browser window.</p></li><li><p>Confirm that the Livechat widget appears on the right pages.</p></li><li><p>Start a chat and check that it appears in Aamu Helpdesk.</p></li><li><p>Answer from Aamu as a Helpdesk user.</p></li><li><p>Confirm that the visitor receives the answer in the widget.</p></li><li><p>End the chat and check that the conversation history remains available.</p></li><li><p>If email fallback is enabled, test the unavailable/busy path too.</p></li><li><p>If AI Livechat is enabled, ask questions that should and should not be answered by AI.</p></li></ol><h2 xmlns="http://www.w3.org/1999/xhtml">Troubleshooting checklist</h2><p xmlns="http://www.w3.org/1999/xhtml">If Livechat does not work as expected, check these:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>The widget snippet is copied from the correct Aamu project.</p></li><li><p>The page loads <code>https://st.aamu.app/livechat.js</code>.</p></li><li><p>The <code>data-host</code> value points to your Aamu subdomain.</p></li><li><p>The <code>data-pid</code> value is the project id from the generated snippet.</p></li><li><p>The website allows the script to load under its content security policy.</p></li><li><p>Helpdesk users have access to the project.</p></li><li><p>At least one support user is ready to answer human chats, unless you are relying on AI or email fallback.</p></li><li><p>Helpdesk email is configured if you expect offline messages to become support work.</p></li><li><p>Team Brain sources are indexed if AI Livechat is enabled.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">When to use Livechat</h2><p xmlns="http://www.w3.org/1999/xhtml">Use Livechat when timing matters: pricing questions, onboarding friction, product questions, sales conversations, and support moments where the visitor is already on your website.</p><p xmlns="http://www.w3.org/1999/xhtml">Use Helpdesk email when the customer can wait. Use AI Livechat when the answer is common and grounded in Team Brain. Use a human agent when the conversation needs judgment. Together, those routes give your support team a single place to handle customer communication without forcing every visitor into the same channel.</p>
