<p xmlns="http://www.w3.org/1999/xhtml">Most AI tools start from a blank chat box. That is useful for asking questions, but it is not always where work happens. In a real workspace, a person is usually looking at a task, a meeting, a Helpdesk ticket, an email thread, a document, or a database row when they know what should happen next.</p><p xmlns="http://www.w3.org/1999/xhtml">Aamu.app is moving AI closer to that moment. Instead of treating AI as only a separate assistant, Aamu can read the current workspace context, understand a short instruction, and route it to a supported action. A comment like <code>@ai Move this meeting to tomorrow at 13:00</code> should not produce a polite suggestion telling someone else to update the meeting. It should update the meeting.</p><p xmlns="http://www.w3.org/1999/xhtml">That is the idea behind AI commands in Aamu: comments, chats, and the AI panel can become small command surfaces for the workspace itself.</p><h2 xmlns="http://www.w3.org/1999/xhtml">From answer bot to action router</h2><p xmlns="http://www.w3.org/1999/xhtml">A plain AI answer is text. An AI command is text plus intent, context, validation, and a product action.</p><p xmlns="http://www.w3.org/1999/xhtml">When a user writes a request, Aamu first decides what kind of request it is. Is the user asking for a general answer? A task draft? A reply draft? A meeting update? A database row change? That intent routing step matters because different requests need different safeguards.</p><p xmlns="http://www.w3.org/1999/xhtml">For example, answering a question from <a target="_blank" rel="noopener noreferrer nofollow" href="/blog/posts/how-aamuapp-uses-team-brain-to-answer-from-your-company-knowledge/">Team Brain</a> is not the same as changing the start time of a meeting. Drafting a Helpdesk reply is not the same as sending it. Creating a new task is not the same as editing an existing database row. Aamu's AI command layer keeps those cases separate.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Where commands can start</h2><p xmlns="http://www.w3.org/1999/xhtml">The command does not have to start in one dedicated AI page. It can come from several places:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>The AI panel</strong>, where the user asks Aamu to work with the current page.</p></li><li><p><strong>Comments with @ai</strong>, where a normal workspace discussion can include an instruction to AI.</p></li><li><p><strong>Chat messages</strong>, where a team conversation can ask AI to create follow-up work or use the current chat as context.</p></li><li><p><strong>Feature-specific AI buttons</strong>, such as Helpdesk triage and support reply drafting.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">This is important because the right context is often already on the screen. A meeting page knows which meeting is being discussed. A database row knows which row and table are open. A Helpdesk ticket knows the customer conversation. A chat knows the recent messages.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What Aamu can do today</h2><p xmlns="http://www.w3.org/1999/xhtml">The current command set focuses on actions that fit naturally inside a team workspace.</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Create task drafts</strong> from the current context, such as a support conversation, email, document, chat, or database row.</p></li><li><p><strong>Update existing tasks</strong> when the task can be resolved from the current page or instruction.</p></li><li><p><strong>Create and update Docs</strong> for internal knowledge and follow-up writing.</p></li><li><p><strong>Draft or rewrite replies</strong> for Helpdesk tickets and email threads while keeping sending as a separate step.</p></li><li><p><strong>Create email drafts</strong> for outbound communication.</p></li><li><p><strong>Create meetings</strong> and update existing meetings.</p></li><li><p><strong>Create databases</strong> and create, update, or link database rows.</p></li><li><p><strong>Search Docs</strong> when the user is asking for existing written knowledge.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">That list is deliberately practical. The goal is not to make every sentence executable. The goal is to support the small actions that interrupt knowledge work all day: make a follow-up task, adjust a meeting, prepare a reply, update a row, connect two records, or turn repeated information into a Doc.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Example: rescheduling a meeting from a comment</h2><p xmlns="http://www.w3.org/1999/xhtml">A meeting comment might say:</p><blockquote xmlns="http://www.w3.org/1999/xhtml"><p>@ai Move this meeting to tomorrow at 13:00</p></blockquote><p xmlns="http://www.w3.org/1999/xhtml">A text-only assistant might reply, "Sure, update the meeting to tomorrow at 13:00." That is not enough. The user already knew what they wanted. The useful action is to change the meeting.</p><p xmlns="http://www.w3.org/1999/xhtml">For this to work well, Aamu has to pass the right context into the command:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>the source type is a meeting, not just a generic page;</p></li><li><p>the existing meeting id, title, start time, end time, and invitees are available;</p></li><li><p>the user's timezone and browser offset are included, so "tomorrow at 13:00" is interpreted as the user's local time;</p></li><li><p>the command result updates the meeting instead of only writing another comment.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">That last point is the product difference. The AI is not only writing about the workspace. It is using the same workspace model that the UI uses.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Why timezone details matter</h2><p xmlns="http://www.w3.org/1999/xhtml">Time commands are deceptively hard. "Tomorrow at 13:00" is clear to a human because the human silently applies local time. A server does not know that unless the client or user profile tells it.</p><p xmlns="http://www.w3.org/1999/xhtml">Aamu sends both the user's timezone name and the browser timezone offset with AI commands where time may be interpreted. The timezone name is useful as a stable user preference. The browser offset is useful because it reflects the user's current environment, including daylight saving time.</p><p xmlns="http://www.w3.org/1999/xhtml">This matters for meeting updates, task dates, reminders, and any future command that turns natural language into a timestamp. Without it, a command can look correct in the AI reply while storing the wrong time.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Example: turning a conversation into a task</h2><p xmlns="http://www.w3.org/1999/xhtml">A common team pattern is that a conversation creates follow-up work. Someone explains a customer issue in a Helpdesk ticket, email thread, chat, or meeting note, and the next step is obvious: make a task for the team.</p><blockquote xmlns="http://www.w3.org/1999/xhtml"><p>@ai Create a task to check why this customer cannot import contacts. Assign it to support and include the relevant details.</p></blockquote><p xmlns="http://www.w3.org/1999/xhtml">The useful task draft should not only contain the sentence the user typed. It should use the current context as source material: the ticket, email thread, chat, or row the user was looking at. It should produce a clear title, useful body, and link back to the original item when possible.</p><p xmlns="http://www.w3.org/1999/xhtml">This is where Aamu's connected workspace helps. Tasks, Docs, Helpdesk, email, meetings, databases, and comments are already part of the same product, so AI does not have to operate from a disconnected browser extension or copied prompt.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Example: editing CRM data from the row itself</h2><p xmlns="http://www.w3.org/1999/xhtml">The new <a target="_blank" rel="noopener noreferrer nofollow" href="/blog/posts/how-to-use-aamuapp-as-a-small-team-crm/">small-team CRM template</a> is a good example of why AI commands become more useful when the data is structured.</p><p xmlns="http://www.w3.org/1999/xhtml">A user might open a deal row and say:</p><blockquote xmlns="http://www.w3.org/1999/xhtml"><p>@ai Move this deal to Proposal, set probability to 60%, and link the primary contact to Anna from Northwind.</p></blockquote><p xmlns="http://www.w3.org/1999/xhtml">That is not just a writing task. It is a database operation. Aamu needs to know the current row, the table columns, the available reference targets, and which fields are safe to change. The command router can then choose between updating the current row, creating a new row, or linking the current row to another row.</p><p xmlns="http://www.w3.org/1999/xhtml">This is the direction where AI starts to feel less like an add-on and more like an editing layer for the workspace.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Human review still matters</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu's AI support features already follow a human-in-the-loop model. A generated Helpdesk or email reply is a draft until someone sends it. That principle still applies when commands become more action-oriented.</p><p xmlns="http://www.w3.org/1999/xhtml">Not every command should be automatic, and not every action has the same risk. Creating a task draft is low risk. Sending a customer reply is higher risk. Updating a meeting time affects other people. Changing CRM data affects reporting and follow-up. A useful AI command system needs product boundaries, not only model confidence.</p><p xmlns="http://www.w3.org/1999/xhtml">That is why Aamu separates intent routing, context gathering, validation, and execution. The AI can interpret the request, but the product still decides what actions exist and how they are applied.</p><h2 xmlns="http://www.w3.org/1999/xhtml">How this fits with Team Brain</h2><p xmlns="http://www.w3.org/1999/xhtml">AI commands are not only about changing objects. Sometimes the right action is to answer from existing company knowledge or find the right internal Doc. That is where <a target="_blank" rel="noopener noreferrer nofollow" href="/blog/posts/ai-support-in-aamuapp/">Aamu's AI support workflow</a> and Team Brain connect to commands.</p><p xmlns="http://www.w3.org/1999/xhtml">A command can use the current page as context and also retrieve related Docs or Team Brain material. That makes the result less generic. A support draft can use the latest policy. A task can include the relevant troubleshooting detail. A Doc update can reflect the source conversation instead of starting from memory.</p><p xmlns="http://www.w3.org/1999/xhtml">The useful pattern is simple: workspace context tells AI what is happening now, and Team Brain tells AI what the team already knows.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Why comments are a natural command surface</h2><p xmlns="http://www.w3.org/1999/xhtml">Comments are already where teams say what should happen next. People write "can someone follow up?", "move this to next week", "turn this into a doc", or "connect this to the customer record" all day long.</p><p xmlns="http://www.w3.org/1999/xhtml">Adding <code>@ai</code> to that flow makes the instruction explicit without moving the user into a separate automation builder. It also keeps the action visible in the same place as the discussion. The team can see that AI was asked to help, and the resulting task, meeting, draft, or row remains connected to the original work.</p><p xmlns="http://www.w3.org/1999/xhtml">This is not a replacement for buttons, forms, or careful workflow design. It is a faster path for the cases where the user can describe the intended action more naturally than they can click through several screens.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The bottom line</h2><p xmlns="http://www.w3.org/1999/xhtml">The most useful AI inside a workspace is not only a chatbot. It is a context-aware layer that can help with the work already in front of the user.</p><p xmlns="http://www.w3.org/1999/xhtml">In Aamu.app, AI commands connect natural language to concrete workspace actions: tasks, Docs, Helpdesk drafts, email drafts, meetings, database rows, and references. The command router decides what kind of request the user made. The current page gives context. Team Brain can add company knowledge. The product executes only the supported action.</p><p xmlns="http://www.w3.org/1999/xhtml">That combination is the interesting part. AI can answer questions, but it can also help keep the workspace moving: update the meeting, create the follow-up, link the CRM row, draft the reply, and leave the team in control.</p>
