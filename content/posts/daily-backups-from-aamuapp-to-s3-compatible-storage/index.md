<p xmlns="http://www.w3.org/1999/xhtml">A team workspace becomes more useful as it gathers real work: tasks, Docs, support conversations, emails, meetings, forms, database rows, CRM records, and internal posts. That also means the workspace becomes more important to protect.</p><p xmlns="http://www.w3.org/1999/xhtml">Aamu.app supports daily team backups to S3-compatible storage. A team can connect its own object storage bucket, enable backups, and let Aamu export changed workspace data as structured JSON files once per day.</p><p xmlns="http://www.w3.org/1999/xhtml">The goal is simple: your team should have a practical copy of its operational data outside the primary application database, in storage you control.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Why S3-compatible backups</h2><p xmlns="http://www.w3.org/1999/xhtml">S3-compatible storage is a useful backup target because it is widely supported. The same basic integration model works with AWS S3 and many compatible platforms such as object storage from cloud providers, infrastructure hosts, and self-hosted S3-compatible systems.</p><p xmlns="http://www.w3.org/1999/xhtml">For a small team, this matters because backups should not require a custom export workflow. The team chooses a bucket, stores credentials in Aamu, and Aamu writes backup objects there on a schedule.</p><p xmlns="http://www.w3.org/1999/xhtml">The bucket can be managed with the team's normal storage policies: retention rules, replication, access logging, lifecycle rules, or whatever the chosen platform supports.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What gets backed up</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu's daily backup focuses on the workspace data that teams usually need if they want to inspect, migrate, audit, or recover operational context.</p><p xmlns="http://www.w3.org/1999/xhtml">The current backup set includes:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Projects</strong>, including basic project metadata and users.</p></li><li><p><strong>Docs</strong>, including searchable document content.</p></li><li><p><strong>Tasks</strong>, including title, content, owner, status, tags, comments, and timestamps.</p></li><li><p><strong>Posts</strong>, including content, author, comments, and timestamps.</p></li><li><p><strong>Helpdesk</strong>, including tickets, chats, chat messages, customers, comments, status, tags, participants, and timestamps.</p></li><li><p><strong>Emails</strong>, including message title, content, recipients, contact, type, comments, and timestamps.</p></li><li><p><strong>Chats</strong>, including rooms and chat messages.</p></li><li><p><strong>Meetings</strong>, including invitees, participants, owner, comments, name, status, tags, start time, and timestamps.</p></li><li><p><strong>Databases</strong>, including databases, tables, columns, and row data.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The backup also follows selected nested objects. For example, a database backup can include its tables and rows. A chat backup can include its chat messages. A Helpdesk chat backup can include the related chat messages.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Incremental by default</h2><p xmlns="http://www.w3.org/1999/xhtml">Daily backups do not need to upload the entire workspace every time. Aamu tracks the latest successful backup timestamp for each backed-up object and uses that to decide which documents have changed.</p><p xmlns="http://www.w3.org/1999/xhtml">When a task, Doc, ticket, row, meeting, or other backed-up object has not changed since the last backup, it can be skipped. When it has changed, Aamu writes a fresh JSON object to the backup bucket.</p><p xmlns="http://www.w3.org/1999/xhtml">This makes the daily run faster and keeps object storage quieter, while still producing readable backup files for changed data.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Readable JSON, organized by object path</h2><p xmlns="http://www.w3.org/1999/xhtml">Backup files are written as JSON, not as an opaque database dump. The object key reflects the Aamu object path.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">projects/PROJECT_ID.json
docs/DOC_ID.json
tasks/TASK_ID.json
helpdesk_tickets/TICKET_ID.json
dbs/DB_ID/dbtables/TABLE_ID/dbrows/ROW_ID.json</code></pre><p xmlns="http://www.w3.org/1999/xhtml">That layout is intentionally boring and inspectable. If someone needs to look at a backed-up task, Doc, ticket, database row, or meeting, the file can be opened directly from object storage.</p><p xmlns="http://www.w3.org/1999/xhtml">It also makes downstream use easier. A backup bucket can feed an audit process, a data lake, a migration tool, or a custom recovery script without first unpacking a monolithic archive.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Media referenced from content</h2><p xmlns="http://www.w3.org/1999/xhtml">Workspace content often includes images or file references. During backup, Aamu can process image references in HTML content and copy those media objects into the backup bucket under a stable media path.</p><p xmlns="http://www.w3.org/1999/xhtml">That means backed-up Docs, posts, tasks, or other HTML content are less dependent on the original media location. The JSON can point to media stored alongside the backup data.</p><h2 xmlns="http://www.w3.org/1999/xhtml">How the storage connection works</h2><p xmlns="http://www.w3.org/1999/xhtml">A backup connection is configured per team. Aamu needs the S3-compatible endpoint, bucket name, access key, secret key, and optionally region and path-style mode.</p><p xmlns="http://www.w3.org/1999/xhtml">Path-style mode is useful for providers that expect bucket names in the URL path instead of as a virtual host. Region is optional because some S3-compatible providers require it and some do not.</p><p xmlns="http://www.w3.org/1999/xhtml">Before writing backup objects, Aamu checks access to the configured bucket. If the bucket cannot be reached, credentials are wrong, or the provider rejects the request, the backup run records an error instead of silently pretending everything worked.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Daily schedule</h2><p xmlns="http://www.w3.org/1999/xhtml">The backup worker runs automatically for enabled teams. On server start it performs a backup run, and then it repeats the backup process every 24 hours.</p><p xmlns="http://www.w3.org/1999/xhtml">Each enabled team's backup is queued during the run. Teams with incomplete storage configuration are skipped until the endpoint, bucket, and credentials are present.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Operational safety details</h2><p xmlns="http://www.w3.org/1999/xhtml">Backup systems should fail loudly enough to be useful. Aamu records backup errors on the team's backup configuration when a run fails. The backup process also limits repeated errors so one broken object or storage problem does not cause an uncontrolled failure loop.</p><p xmlns="http://www.w3.org/1999/xhtml">Uploads are logged with progress during larger runs. If there are no changed objects, the run completes without uploading new JSON files.</p><p xmlns="http://www.w3.org/1999/xhtml">A full backup can also reset the stored backup timestamps for a team, which forces Aamu to treat existing objects as needing to be exported again.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What this is not</h2><p xmlns="http://www.w3.org/1999/xhtml">Daily S3-compatible backups are not a replacement for every reliability layer. They do not remove the need for primary database backups, application monitoring, access control, or retention planning.</p><p xmlns="http://www.w3.org/1999/xhtml">They serve a different purpose: giving the team an application-level, structured, portable copy of important workspace data in external object storage.</p><p xmlns="http://www.w3.org/1999/xhtml">That distinction matters. A database snapshot is useful for restoring infrastructure. A structured JSON backup is useful when a team wants to inspect its work, move data, recover specific records, or keep a long-lived archive outside the main application.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The bottom line</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu.app's daily backup feature is built for a practical kind of data ownership. Enable a team backup, point it at an S3-compatible bucket, and Aamu exports changed workspace data as readable JSON files every day.</p><p xmlns="http://www.w3.org/1999/xhtml">For teams using Aamu for tasks, Docs, support, email, meetings, forms, databases, and CRM work, that creates a second copy of the operational memory they are building inside the workspace.</p>
