<p xmlns="http://www.w3.org/1999/xhtml">Aamu.app can be used as a lightweight headless CMS for a static blog. This official blog is built that way: the post metadata lives in an Aamu Database, the article text is written in Aamu Docs, and a build script turns everything into a Hugo site.</p><p xmlns="http://www.w3.org/1999/xhtml">The result is a practical publishing workflow. Writers can use Aamu Docs, editors can manage status and metadata in the Database, and the public site stays fast because Hugo generates static HTML that can be hosted almost anywhere.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What we are building</h2><p xmlns="http://www.w3.org/1999/xhtml">The basic flow looks like this:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">Aamu Docs
  -&gt; BlogPost row in Aamu Database
  -&gt; GraphQL API + Docs API
  -&gt; Hugo content files
  -&gt; static blog</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The Database row is used for the structured parts of the post: title, slug, author, publish date, status, tags, hero image, and the linked Doc. The Doc is the source of truth for the article body. Hugo is used only at build time to create the public site.</p><p xmlns="http://www.w3.org/1999/xhtml">You can see the source code for this blog at <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/AamuApp/aamu-blog">https://github.com/AamuApp/aamu-blog</a>.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Why use Aamu.app as a headless CMS?</h2><p xmlns="http://www.w3.org/1999/xhtml">A traditional CMS can be useful, but it also becomes another product to maintain. If your team already works in Aamu, using Aamu as the CMS keeps writing, metadata, project work, and internal knowledge close together.</p><p xmlns="http://www.w3.org/1999/xhtml">This approach is useful when you want:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>a familiar writing experience through Aamu Docs,</p></li><li><p>structured blog metadata in an Aamu Database,</p></li><li><p>a public site that is static, fast, and easy to host,</p></li><li><p>a build pipeline that can run on your own server, CI, or a static hosting service,</p></li><li><p>GraphQL access to the database rows, and</p></li><li><p>Docs API access to the actual article HTML.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">It also gives you a nice separation of responsibilities. Aamu is where the content is written and managed. Hugo is where the public site is generated. Your hosting provider only needs to serve static files.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The ingredients</h2><p xmlns="http://www.w3.org/1999/xhtml">For a setup like this blog, you need:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>an Aamu Database created from the <code>aamu-blog</code> template,</p></li><li><p>Aamu Docs for writing the post bodies,</p></li><li><p>a BlogPost table where each row stores post metadata and links to a Doc,</p></li><li><p>a Person table for authors,</p></li><li><p>an API key that can read the database and the linked Docs,</p></li><li><p>Hugo for generating the static site, and</p></li><li><p>some deployment target such as Netlify, Cloudflare Pages, Vercel, or your own server.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">Create the blog database</h2><p xmlns="http://www.w3.org/1999/xhtml">Create a database in Aamu.app with the provided <code>aamu-blog</code> template. The template gives you the basic shape for a blog: a BlogPost table and a Person table.</p><p xmlns="http://www.w3.org/1999/xhtml">The BlogPost table is for posts. It stores fields such as title, slug, description, publish date, status, tags, author, hero image, and the Doc reference. The Person table is for post authors.</p><p xmlns="http://www.w3.org/1999/xhtml">The important point is that the post body does not need to live in a plain database text field. The preferred workflow is to write the article in Aamu Docs and link that Doc to the BlogPost row.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Write posts in Aamu Docs</h2><p xmlns="http://www.w3.org/1999/xhtml">Create a new Doc for the article and write the post there. This gives you the normal Aamu Docs writing experience: rich text, headings, lists, links, images, collaboration, snapshots, and source views.</p><p xmlns="http://www.w3.org/1999/xhtml">When the post is ready, create or update the matching BlogPost row in the database. Set the title, slug, author, publish date, tags, hero image, and status. Then select the Doc in the row's Doc field.</p><p xmlns="http://www.w3.org/1999/xhtml">That Doc is the article source. The build script reads the BlogPost rows through GraphQL and then fetches the linked Doc through the Docs API. If the Doc has HTML content, that HTML becomes the Hugo post body.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What about the Body field?</h2><p xmlns="http://www.w3.org/1999/xhtml">The Body field is only for manually written or copy-pasted HTML. It is useful as a fallback, or if you really want to manage the post body as raw HTML in the database row.</p><p xmlns="http://www.w3.org/1999/xhtml">Do not use the Body field as the normal Doc workflow. In the current Aamu.app UI, you cannot attach a Doc to the Body field. If you want the post to be written in Docs, link the Doc through the row's Doc field instead.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Configure the API key</h2><p xmlns="http://www.w3.org/1999/xhtml">The build script needs to read published BlogPost rows and fetch the linked Docs. This is done with an API key stored as an environment variable on the machine or CI job that builds the site.</p><p xmlns="http://www.w3.org/1999/xhtml">For this blog repository, the useful environment variables are:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">API_KEY=your_api_key
AAMU_DB_ID=your_database_id
AAMU_API_BASE_URL=https://your-aamu-host.example

# Optional, depending on your setup:
DOCS_API_KEY=your_docs_api_key
AAMU_PROJECT_ID=your_project_id
GRAPHQL_ENDPOINT=https://api.aamu.app/api/v1/graphql/</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Keep API keys on the server side. Do not commit them to Git, do not publish them in frontend JavaScript, and do not paste them into public examples. The static site can be public, but the build credentials should stay private.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Build the Hugo site</h2><p xmlns="http://www.w3.org/1999/xhtml">The blog repository contains a build script that does the content sync before Hugo runs. The basic command is:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">node build-with-graphql.js
hugo --minify</code></pre><p xmlns="http://www.w3.org/1999/xhtml">Or as one command:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">node build-with-graphql.js &amp;&amp; hugo --minify</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The script does a few things:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>queries the BlogPost rows through the GraphQL API,</p></li><li><p>filters for published posts,</p></li><li><p>detects the Doc field in the BlogPost schema,</p></li><li><p>fetches the linked Doc through the Docs API,</p></li><li><p>uses the Doc HTML as the post body,</p></li><li><p>downloads images into the Hugo post folder,</p></li><li><p>writes Hugo-compatible <code>content/posts/.../index.md</code> files, and</p></li><li><p>lets Hugo create the final static site.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The generated Hugo content files are build artifacts. The source of truth is the Aamu Database row plus the linked Doc.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Deployment options</h2><p xmlns="http://www.w3.org/1999/xhtml">There are several reasonable ways to publish the generated static site.</p><h3 xmlns="http://www.w3.org/1999/xhtml">Static hosting service</h3><p xmlns="http://www.w3.org/1999/xhtml">You can use a service such as Netlify, Cloudflare Pages, Vercel, or another static host. In that setup, configure the environment variables in the hosting service and use the build command above. A build hook can trigger a new build when content changes.</p><h3 xmlns="http://www.w3.org/1999/xhtml">Your own server</h3><p xmlns="http://www.w3.org/1999/xhtml">You can also build on your own machine or server and serve the generated files with Nginx or another web server. The official Aamu blog has used a Git-based deployment flow with a bare repository and a post-receive hook. That works well if you like controlling the server yourself, but it is only one option.</p><h3 xmlns="http://www.w3.org/1999/xhtml">CI pipeline</h3><p xmlns="http://www.w3.org/1999/xhtml">A third option is a CI job. The job checks out the blog repository, sets the Aamu API environment variables, runs the build, and uploads the generated <code>public/</code> directory to your hosting target.</p><h2 xmlns="http://www.w3.org/1999/xhtml">How this official blog is built</h2><p xmlns="http://www.w3.org/1999/xhtml">This post itself is written with Aamu Docs. Its metadata is stored in the Aamu blog database. The public page you are reading is generated by Hugo from that data.</p><p xmlns="http://www.w3.org/1999/xhtml">That makes the blog a small but real example of Aamu's "one workspace" idea. The same product contains the writing tool, the database, the API surface, and the internal workflow around the content. Hugo and the hosting provider handle the final public website.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Next steps</h2><p xmlns="http://www.w3.org/1999/xhtml">If you want to build a blog like this:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Fork or clone <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/AamuApp/aamu-blog">AamuApp/aamu-blog</a>.</p></li><li><p>Create an Aamu Database from the <code>aamu-blog</code> template.</p></li><li><p>Create a Person row for the author.</p></li><li><p>Create a Doc for your first post.</p></li><li><p>Create a BlogPost row and link the Doc in the row's Doc field.</p></li><li><p>Add the API environment variables to your build environment.</p></li><li><p>Run <code>node build-with-graphql.js &amp;&amp; hugo --minify</code>.</p></li><li><p>Deploy the generated <code>public/</code> directory.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">After that, publishing a post is mostly content work: write in Docs, update the BlogPost metadata, set the status to published, and run the build.</p>
