<p xmlns="http://www.w3.org/1999/xhtml">A workspace becomes much more useful when it can tell other systems what just happened. A new Helpdesk ticket should be able to open an incident workflow. A task assignment should be able to notify an external planning tool. A submitted form should be able to create a lead, update a database, or call a custom automation.</p><p xmlns="http://www.w3.org/1999/xhtml">That is why Aamu.app now supports outbound webhooks. A team can register an endpoint URL, choose the events it wants to receive, optionally narrow the webhook to selected projects, and let Aamu send signed HTTP requests when matching workspace activity happens.</p><p xmlns="http://www.w3.org/1999/xhtml">The first version is intentionally practical: server-side delivery, signed payloads, visible delivery logs, and event coverage across the parts of Aamu where integrations usually start.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What webhooks are for</h2><p xmlns="http://www.w3.org/1999/xhtml">An API is useful when another system asks Aamu for data. A webhook is useful when Aamu tells another system about a change.</p><p xmlns="http://www.w3.org/1999/xhtml">For example, an integration can listen for <code>helpdesk.ticket.created</code> and immediately start a triage workflow. Another can listen for <code>task.status_changed</code> and update an external reporting dashboard. A third can listen for <code>form.submitted</code> and send the response to a CRM or internal queue.</p><p xmlns="http://www.w3.org/1999/xhtml">The important difference is timing. Instead of polling Aamu every few minutes and asking "anything new?", the receiving system gets called when the event happens.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Team-level setup with project filters</h2><p xmlns="http://www.w3.org/1999/xhtml">Aamu webhooks are configured at the team level. That means the webhook belongs to the workspace team, not to a single browser session or one local client.</p><p xmlns="http://www.w3.org/1999/xhtml">Many Aamu events are project-specific, so webhook definitions can also include project filters. If a webhook is limited to one project, it receives only matching events from that project. If it has no project filter, it can receive matching events from across the team.</p><p xmlns="http://www.w3.org/1999/xhtml">Some events, such as team member changes, are naturally team-level and do not have a project id. Project filters apply when the event itself has project context.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Events Aamu can send</h2><p xmlns="http://www.w3.org/1999/xhtml">The current webhook event list covers the main operational objects in Aamu:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><strong>Helpdesk</strong>: tickets, status changes, assignment, comments, email replies, and messages.</p></li><li><p><strong>Tasks</strong>: created, updated, status changes, assignment changes, due date changes, and comments.</p></li><li><p><strong>Forms</strong>: created, updated, deleted, and submitted.</p></li><li><p><strong>Email</strong>: received, sent, and replied.</p></li><li><p><strong>Meetings</strong>: created, updated, deleted, and comments.</p></li><li><p><strong>Files</strong>: created, updated, deleted, and comments.</p></li><li><p><strong>Projects and teams</strong>: project lifecycle events, project member changes, and team member role changes.</p></li><li><p><strong>Docs</strong>: created, updated, deleted, and comments.</p></li><li><p><strong>Databases</strong>: databases, tables, and rows being created, updated, or deleted.</p></li><li><p><strong>Posts</strong>: created, updated, deleted, and comments.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">The event names are explicit, such as <code>task.created</code>, <code>helpdesk.ticket.assigned</code>, <code>email.received</code>, <code>meeting.updated</code>, <code>db.row.created</code>, and <code>project.member.added</code>. This makes it easy for the receiving service to route events by type.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Forms and database-row events</h2><p xmlns="http://www.w3.org/1999/xhtml">Forms have lifecycle events for the Form itself:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p><code>form.created</code> when a Form is created.</p></li><li><p><code>form.updated</code> when its questions, publishing state, or settings are changed.</p></li><li><p><code>form.deleted</code> when it is deleted.</p></li><li><p><code>form.submitted</code> when someone sends a response through a public Form or the authenticated Forms API.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">A published Form stores each response as a row in its connected database table. One submission therefore also produces <code>db.row.created</code>. These two events serve different integration needs: <code>form.submitted</code> contains Form and submission context, normalized field values, and uploaded-file metadata, while <code>db.row.created</code> represents the new database row.</p><p xmlns="http://www.w3.org/1999/xhtml">If that response row is edited later, Aamu emits <code>db.row.updated</code>. Database-row events are also emitted for rows created or changed through the database UI, Forms, the authenticated API, or GraphQL, so an integration can listen at the Form level or at the underlying database level.</p><p xmlns="http://www.w3.org/1999/xhtml">A CRM integration might subscribe to <code>form.submitted</code> to ingest a new lead with its submitted fields. A more general data-sync integration might instead subscribe to <code>db.row.created</code> and <code>db.row.updated</code> for the Form's backing project and then use the Aamu API to fetch any additional row context it needs.</p><h2 xmlns="http://www.w3.org/1999/xhtml">What a delivery looks like</h2><p xmlns="http://www.w3.org/1999/xhtml">When an event matches an enabled webhook, Aamu sends an HTTP <code>POST</code> request to the configured URL. The request body is JSON and includes the event id, event type, creation time, team id, optional project id, actor id, and event-specific data.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-json">{
  "id": "evt_92ac2f69-6dbd-4ced-ac0a-dbc793396571",
  "type": "task.created",
  "createdAt": "2026-06-05T14:47:14.806Z",
  "teamId": "TEAM_ID",
  "projectId": "PROJECT_ID",
  "actorId": "USER_ID",
  "data": {
    "task": {
      "id": "TASK_ID",
      "title": "Follow up with the customer",
      "status": "active"
    }
  }
}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The request also includes headers that help the receiver identify and verify the delivery:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">user-agent: Aamu-Webhooks/1.0
x-aamu-delivery-id: whd_c5784f35-59bc-48e7-9017-d4447d9ed631
x-aamu-event-id: evt_92ac2f69-6dbd-4ced-ac0a-dbc793396571
x-aamu-event-type: task.created
x-aamu-signature: t=1780670836831,v1=f3056005aa0deb8d50a96156ed7dcc8a9ed73dda1042466c72a086a5f568a93c</code></pre><h2 xmlns="http://www.w3.org/1999/xhtml">Signed requests</h2><p xmlns="http://www.w3.org/1999/xhtml">Webhook requests are signed with a shared secret. The receiving service should verify the <code>x-aamu-signature</code> header before trusting the request.</p><p xmlns="http://www.w3.org/1999/xhtml">The header contains a timestamp and a versioned HMAC SHA-256 signature. A receiver can rebuild the signed payload from the timestamp and raw request body, calculate the expected HMAC with the webhook secret, and compare it to the <code>v1</code> value.</p><p xmlns="http://www.w3.org/1999/xhtml">This protects the integration from accepting random HTTP requests that merely look like Aamu webhook deliveries.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Logs for both success and failure</h2><p xmlns="http://www.w3.org/1999/xhtml">Webhook development is much easier when failures are visible. Aamu records webhook events and delivery attempts so the team can see what was sent, where it was sent, whether it succeeded, how long it took, and what error happened when delivery failed.</p><p xmlns="http://www.w3.org/1999/xhtml">That visibility matters because webhook failures are often environmental. The URL might be unreachable from the server. A local test server might not be running. The receiver might return a non-2xx status. A signing secret might be missing. Without delivery logs, all of those problems feel the same from the product UI: nothing happened.</p><p xmlns="http://www.w3.org/1999/xhtml">With logs, the integration builder can see the difference between "the event was not emitted", "the event was emitted but did not match this webhook", "the delivery failed before sending", and "the receiver returned an error".</p><h2 xmlns="http://www.w3.org/1999/xhtml">Testing with a local endpoint</h2><p xmlns="http://www.w3.org/1999/xhtml">A simple way to test a webhook is to run a local HTTP server that prints incoming requests and then configure an Aamu webhook to call it. During development, the receiver can log headers and body so the integration author sees the exact payload shape.</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">--- webhook received ---
POST /webhook
x-aamu-event-type: task.created
x-aamu-delivery-id: whd_...
x-aamu-signature: t=...,v1=...

{
  "id": "evt_...",
  "type": "task.created",
  "teamId": "TEAM_ID",
  "projectId": "PROJECT_ID",
  "data": { "task": { "id": "TASK_ID", "title": "Test task" } }
}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">For local machine testing, remember that the Aamu server must be able to reach the URL. If Aamu is running somewhere else, <code>127.0.0.1</code> points to that server's own machine, not to the developer's laptop. In that case, use a reachable tunnel or a test endpoint hosted where Aamu can call it.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Example integrations</h2><p xmlns="http://www.w3.org/1999/xhtml">The first webhook use cases are straightforward but powerful:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Create an external incident when an urgent Helpdesk ticket is created.</p></li><li><p>Send task changes to a reporting database or data warehouse.</p></li><li><p>Push form submissions into a CRM or sales workflow.</p></li><li><p>Notify an operations channel when a database row is created or updated.</p></li><li><p>Start a document review workflow when an important Doc changes.</p></li><li><p>Sync project membership changes to an internal access-management process.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">Because webhooks carry both team and project context, the receiving service can decide whether to treat the event as a workspace-wide automation, a project-specific workflow, or a customer-specific integration.</p><h2 xmlns="http://www.w3.org/1999/xhtml">How this fits with the Aamu API</h2><p xmlns="http://www.w3.org/1999/xhtml">Webhooks and the Aamu API work well together. The webhook tells the integration that something changed. The API can then fetch more detail, create follow-up work, update a Doc, write a Helpdesk draft, or send data into another part of Aamu.</p><p xmlns="http://www.w3.org/1999/xhtml">A common pattern is:</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>Receive a webhook event.</p></li><li><p>Verify the signature.</p></li><li><p>Use the event type and ids to choose a workflow.</p></li><li><p>Call the Aamu API if more context or a follow-up action is needed.</p></li><li><p>Store the delivery id so repeated deliveries can be handled idempotently.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">That separation keeps webhook payloads focused while still allowing integrations to do richer work when they need it.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The bottom line</h2><p xmlns="http://www.w3.org/1999/xhtml">Outbound webhooks make Aamu easier to connect to the rest of a team's tools. Tasks, Helpdesk tickets, email, forms, meetings, files, docs, databases, posts, projects, and team membership can now become real-time integration triggers.</p><p xmlns="http://www.w3.org/1999/xhtml">The design is deliberately boring in the best sense: server-side delivery, explicit event names, signed requests, project filtering, and delivery logs. Those are the pieces an integration needs before it can be trusted in everyday work.</p>
