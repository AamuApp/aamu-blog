<p xmlns="http://www.w3.org/1999/xhtml">A workspace assistant is most useful when it understands the difference between asking for product help and asking it to change the workspace. A user who writes “How do I share this Doc?” needs an explanation. A user who writes “Create a task from this Doc” wants an action.</p><p xmlns="http://www.w3.org/1999/xhtml">Aamu.app now has a dedicated AI product-help path for the first case. In the Launch AI dialog, start a question with <code>aamu:</code> to ask how Aamu works. The answer is generated from Aamu's public documentation and includes the source pages used to prepare it.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Using aamu: in Launch AI</h2><p xmlns="http://www.w3.org/1999/xhtml">Open Launch AI from the Aamu header and write a question such as:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">aamu: How do I share an Aamu Doc with a public link?</code></pre><p xmlns="http://www.w3.org/1999/xhtml">You can ask in your own language:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">aamu: Miten teen tietokanta-automaation, joka käynnistyy kun uusi rivi lisätään?</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The <code>aamu:</code> prefix routes the question to Aamu's product-help assistant before the normal workspace action router. This keeps the request separate from commands that create tasks, edit Docs, update meetings, draft replies, or change database rows.</p><p xmlns="http://www.w3.org/1999/xhtml">The current Aamu page path is included as lightweight context. A question such as <code>aamu: How do I automate this?</code> can therefore be easier to interpret when the user is already looking at a database or project settings page. Team content from the current workspace is not sent into the public documentation search.</p><h2 xmlns="http://www.w3.org/1999/xhtml">A separate knowledge base for Aamu itself</h2><p xmlns="http://www.w3.org/1999/xhtml">Every Aamu team can have its own Team Brain. A team's Team Brain contains knowledge relevant to that organization: internal Docs, policies, support material, selected web pages, FAQs, and other maintained sources.</p><p xmlns="http://www.w3.org/1999/xhtml">Product help needs a different knowledge base. Questions about Aamu should be answered from Aamu's own public material, not from the customer's Team Brain. The product-help assistant therefore uses a dedicated Aamu support workspace whose Team Brain indexes the Aamu landing page and the public Aamu blog sitemap.</p><p xmlns="http://www.w3.org/1999/xhtml">This separation has two benefits:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Aamu product instructions do not become mixed with the customer's company knowledge.</p></li><li><p>The product-help service can be available even when the customer's own team AI is not enabled or does not have a connected model account.</p></li></ul><h2 xmlns="http://www.w3.org/1999/xhtml">What the assistant can answer</h2><p xmlns="http://www.w3.org/1999/xhtml">The quality and coverage follow the public documentation. Current material can help with questions about areas such as:</p><ul xmlns="http://www.w3.org/1999/xhtml"><li><p>Team Brain knowledge sources, indexing, and retrieval;</p></li><li><p>AI-assisted Helpdesk and email reply drafts;</p></li><li><p>public sharing and snapshots in Aamu Docs;</p></li><li><p>tasks, Forms, meetings, files, and project workflows;</p></li><li><p>database tables, rows, GraphQL, and database automations;</p></li><li><p>the Aamu API, API keys, scopes, and supported operations;</p></li><li><p>AI commands and the distinction between drafting, acting, and sending.</p></li></ul><p xmlns="http://www.w3.org/1999/xhtml">If the indexed public material does not support a reliable answer, the assistant should say so instead of inventing a product capability. For example, an undocumented interface setting should not be presented as available merely because a general AI model considers it plausible.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Answers include public sources</h2><p xmlns="http://www.w3.org/1999/xhtml">Before writing an answer, the service searches the dedicated support Team Brain for relevant entries and document chunks. Only results connected to public Aamu URLs are allowed into the answer context.</p><p xmlns="http://www.w3.org/1999/xhtml">The response includes a list of source pages. These links make the answer easier to inspect and give the user a path to the longer product explanation. A sitemap may contain many blog posts, but an individual answer shows only a limited set of URLs selected from the relevant retrieval results.</p><p xmlns="http://www.w3.org/1999/xhtml">The source list is supporting context, not a claim that every linked page contributed equally to every sentence. As the documentation and retrieval metadata improve, source attribution can become more precise.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The public Support API</h2><p xmlns="http://www.w3.org/1999/xhtml">The same product-help service is available as a small public HTTP API:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">POST https://support.aamu.app/api/ask
Content-Type: application/json

{
  "question": "How do I add a sitemap to Team Brain?",
  "language": "en",
  "context": {
    "path": "/project/edit/PROJECT_ID/helpdesk"
  }
}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">A response contains the answer, public sources, a retrieval confidence value, and an indication of whether supporting material was found:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">{
  "answer": "Add the sitemap as a Team Brain knowledge source...",
  "sources": [
    {
      "title": "Team Brain is Aamu's shared knowledge layer",
      "url": "https://aamu.app/blog/posts/..."
    }
  ],
  "found": true,
  "confidence": 0.61
}</code></pre><p xmlns="http://www.w3.org/1999/xhtml">The API does not require an API key. The hostname fixes the service to the Aamu support knowledge base, and callers cannot select another team, project, or Team Brain through request parameters.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Why the API is public but constrained</h2><p xmlns="http://www.w3.org/1999/xhtml">The indexed knowledge consists of public Aamu pages, so requiring every caller to create an API key would add friction without protecting private workspace content. The endpoint is therefore intentionally public.</p><p xmlns="http://www.w3.org/1999/xhtml">Public does not mean unlimited. The service applies request-size limits, question and context length limits, an IP-based rate limit, output limits, and a short response cache. It is read-only: it cannot create a support ticket, contact a person, send a message, or modify an Aamu workspace.</p><p xmlns="http://www.w3.org/1999/xhtml">The endpoint also rejects requests made through another Aamu team hostname. <code>POST /api/ask</code> is specifically a service of <code>support.aamu.app</code>, not a generic unauthenticated path into customer Team Brains.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Product help is not a human support channel</h2><p xmlns="http://www.w3.org/1999/xhtml">This feature is AI product help. It does not escalate a conversation to a person, promise a human reply, or open a ticket. That boundary is deliberate: the Launch AI dialog is a workspace assistant, not a hidden Helpdesk queue.</p><p xmlns="http://www.w3.org/1999/xhtml">The assistant is best for questions that can be answered from maintained product documentation. Account-specific problems, billing decisions, security reports, and situations requiring investigation still need an appropriate human contact path.</p><h2 xmlns="http://www.w3.org/1999/xhtml">Examples</h2><p xmlns="http://www.w3.org/1999/xhtml">Here are some questions that fit the product-help path:</p><pre xmlns="http://www.w3.org/1999/xhtml"><code class="language-plaintext">aamu: How do I add a Doc as a Team Brain source?

aamu: How does an AI Helpdesk reply draft reach the reply editor?

aamu: How do I share a Doc as a read-only snapshot?

aamu: How do I create a row_inserted database automation?

aamu: Which permissions does an API key need to retrieve Team Brain knowledge?</code></pre><p xmlns="http://www.w3.org/1999/xhtml">A question can also expose a documentation gap. If the assistant can describe an API operation but cannot explain the equivalent UI workflow, that is useful feedback: the public product documentation needs a clearer step-by-step guide.</p><h2 xmlns="http://www.w3.org/1999/xhtml">A documentation loop</h2><p xmlns="http://www.w3.org/1999/xhtml">The product-help assistant turns Aamu's public documentation into an interactive interface. It also makes documentation quality easier to evaluate.</p><ol xmlns="http://www.w3.org/1999/xhtml"><li><p>A user asks a real product question.</p></li><li><p>The support Team Brain retrieves the most relevant public material.</p></li><li><p>The assistant answers from that material and returns links.</p></li><li><p>A weak or incomplete answer reveals missing, vague, or poorly structured documentation.</p></li><li><p>The documentation is improved and the support Team Brain is indexed again.</p></li></ol><p xmlns="http://www.w3.org/1999/xhtml">This is the same principle that makes Team Brain useful for customer support: better maintained knowledge produces better grounded answers. In this case, Aamu itself is the product being supported.</p><h2 xmlns="http://www.w3.org/1999/xhtml">The bottom line</h2><p xmlns="http://www.w3.org/1999/xhtml">Use <code>aamu:</code> in Launch AI when the question is about using Aamu.app. The request goes to a dedicated read-only product-help assistant, searches Aamu's public documentation, and returns an answer with supporting links.</p><p xmlns="http://www.w3.org/1999/xhtml">For integrations, the same capability is available without an API key at <code>POST https://support.aamu.app/api/ask</code>. It is a small, focused API: one support knowledge base, public sources, no workspace actions, and no pretend human handoff.</p>
